## 1. Tool Restructure (User Decision)

### navigate_to: Main movement tool (NEW)
- LLM sees map via inspect_surroundings → decides where to go → calls navigate_to
- C# uses AutoAct pathfinder internally
- Interrupts on: enemy visible, damage taken, hazard, popup
- Replaces explore-type candidates (auto_explore, move_to_waypoint, move_to_stairs)

### choose: Direct decision tool for non-combat modals (NEW)
- Handles: dialogue choices, level-up options, confirmation popups, merchant (Phase 5+)
- LLM sees choices via inspect_surroundings or check_status → calls choose(choice_id, reason)
- No SafetyGate needed (these choices can't kill you)
- Replaces request_candidates for dialogue/level_up/modal situations

### request_candidates + execute: Combat only
- request_candidates situation enum: remove "explore", "dialogue", "level_up", "modal"
- Keep only "combat" (possibly rename to just request_combat_options)
- SafetyGate BLOCK/WARN/PASS applies only here

### get_help → structured assessment tools (User Decision)
- Pro: "get_help is a hidden policy engine — split it"
- Split into: assess_threat, assess_escape, assess_ability (+ possibly more)
- Each returns structured data, not free-form natural language
- Policy boundary becomes explicit: each tool has a clear, auditable responsibility

### Candidate score visibility (User Decision)
- LLM sees: desc, tag, safety — but NOT score values
- Candidate order is randomized (prevents positional bias)
- Rationale: prevents anchor bias; LLM should decide based on situation, not C# scoring
- Fallback still uses highest score (this is C#'s decision, not LLM's)

### SafetyGate multi-layer (User Decision)
- **Pre-filter on navigate_to**: WARN on dangerous zone (zone_tier > player_level/3)
- **Pre-filter on choose**: WARN on irreversible dialogue options (if detectable)
- **Post-validation on execute**: BLOCK/WARN/PASS (unchanged)
- Aligns with Pro's recommendation: "pre-filter + post-validation, not single gate"

### Revised tool inventory (12-13 tools)

| Tool | Purpose | LLM Agency | SafetyGate |

| inspect_surroundings | See map, entities, terrain | Free | — |

## 3. Spatial Memory (User Decision)

### C#-built spatial log in Layer 2 Auto-Log
- C# automatically records zone visits, discovered features, known dangers per zone
- Stored in SQLite: zone_id, visit_count, last_visited_turn, enemies_seen, items_found, danger_level
- LLM queries via get_help("what do I know about this zone?")
- Separate from zone_notes (which is LLM's subjective notes)

### New SQLite table
```sql
CREATE TABLE spatial_log (
    zone_id TEXT PRIMARY KEY,
    zone_name TEXT,
    zone_tier INTEGER,
    visit_count INTEGER DEFAULT 1,
    last_visited_turn INTEGER,
    enemies_encountered TEXT,  -- JSON array of archetype names
    items_found TEXT,          -- JSON array
    danger_rating TEXT,        -- "safe" | "moderate" | "dangerous" | "deadly"
    notes TEXT,                -- auto-generated summary
    updated_at TEXT NOT NULL DEFAULT (datetime('now'))
);
```

## 10. Self-Improvement Loop Refinements (Research + User Decision)

### Design principle: Issue-based semi-automatic loop

The self-improvement loop works via **structured issue proposals**, not free-form reasoning.
LLM analyzes aggregated data across N runs, proposes a concrete change, human approves/rejects.

### Timing: N-run aggregation, not per-death

- **Per-death**: Generate structured death recap → store to SQLite + inject into next run's notes.
  This is the existing Level 3 design. No improvement proposals here.
- **Per-N-runs (N=3-5)**: LLM receives SQLite aggregation queries (not raw logs) and proposes
  improvement issues. This is the self-improvement loop.

Single-run analysis is too noisy (RNG, map variance). Only repeated patterns across
multiple runs warrant improvement proposals.

### Data fed to meta-agent (structured summaries only)

```
What the meta-agent receives:
├── Death cause distribution (SQL: SELECT root_cause_key, COUNT(*) FROM death_events ...)
├── Tier 2 strategic error frequency (SQL: top strategic errors across N runs)
├── Encounter win/loss rates by enemy type
├── SafetyGate WARN/BLOCK frequency and reasons
├── Note drift analysis (what was written/overwritten most)
├── Key decision points (moments where SafetyGate issued WARN)
└── Candidate selection patterns (always picks c1? ignores defensive options?)

What the meta-agent does NOT receive:
├── Raw turn-by-turn logs
├── Full tool call histories
└── Full state snapshots
```

### Issue template (structured, no free-form reasoning)

```markdown
## Observed pattern (from SQL aggregation)
- Death cause: SWARM × 4/5 runs
- Common condition: HP<40%, adjacent>=3, retreat candidates existed but not selected

## Proposal category (select one)
☐ SafetyGate threshold adjustment
☐ Tool description / schema modification
☐ System prompt modification
☐ Scoring weight adjustment
☐ New C# observation needed (information not currently exposed to LLM)
☐ Other (requires human judgment)

## Specific change
- safety_gate.flee_warn_threshold: HP 40% → 50%

## Verification method
- Fixed-seed replay: SWARM death count should decrease
- Expected metric: flee_rate in SWARM encounters increases
```

### Allowed proposal types

| Type | Example | Why it works |
|------|---------|-------------|
| Parameter adjustment | Flee threshold HP 40%→50% | Fixed-seed replay verifiable |
| C# info request | "Expose enemy range to inspect_surroundings" | Death log shows "died to ranged without knowing range" |
| Tool description fix | "Add threat_level explanation to inspect_surroundings docs" | Tool usage pattern measurably changes |
| Score weight tuning | "enemies_killed weight 15→10 (over-rewards risky engagement)" | Survival rate on fixed seeds |
| Prompt tweak | "Add 'check HP before engaging' to combat instructions" | Heal-before-combat rate measurable |

### NOT allowed (auto-reject)

- Vague strategy changes ("explore more carefully")
- Architectural proposals ("add a new tool for X")
- CandidateGenerator logic changes
- Anything without a verification method

### Realistic scope for Appendix A
- Phase 5 initial: **semi-automatic only** (meta-agent proposes, human approves)
- Automation limited to: tool description edits, safety threshold adjustments, system prompt tweaks
- NOT automated: CandidateGenerator logic changes, new tool creation, architectural changes

### Required additions to logging for meta-agent
- candidate_scores (all candidates per turn, not just selected)
- safety_gate_decisions (what was blocked/warned and why)
- aggregation queries for meta-agent context (not raw logs)

### Plateau detection

- Definition: **fixed-seed replay shows same behavioral pattern** (not just survival_turns stagnation)
- Trigger: 3 consecutive improvement cycles with no measurable change on eval suite
- Response: escalate to human with full aggregation report
- NOT based on survival_turns alone (avoids flee-bot trap)

### Failure modes to document
1. Local patch repetition (threshold tweaking → stagnation)
2. Conservatism spiral (too many failures → stops proposing changes)
3. Evaluation metric trap (optimizing survival_turns only → flee-bot)
4. Generation-verification gap (can propose but can't evaluate if improvement is real)
5. Plausible-but-wrong causal attribution (LLM explains "why" incorrectly)

## 11. Pro Independent Review Findings (v3.1, chatId: 69b9d27f)

**Verdict**: "強い設計メモ。最終仕様書ではない。"

### Must-fix spec bugs (HIGH, from Pro)

- [ ] **`previous_response_id` does not inherit `instructions`**
  Continuation calls after the first in tool_loop.py drop the system prompt.
  Fix: re-send `instructions` on every continuation, or pin via separate layer.

- [ ] **`execute` missing `candidate_set_id` + `snapshot_hash` params**
  Stale candidate problem: if candidates are re-requested within a turn,
  `candidate_id` alone is ambiguous. Add both as required args; mismatch = hard-fail.

- [ ] **Cross-run seeding contradiction**
  §4 says "no single best run seed" but Layer 3 + Phase 3 say "best run's notes seed next run".
  Unify to: "only entries confirmed in ≥2/5 runs with confirm > contradict carry forward."

- [ ] **`request_candidates("auto")` in fallback code**
  Already covered in §6 HIGH — remove "auto", use "combat" directly.

### New design issues (from Pro)

- [ ] **get_help is too broad** — "hidden policy engine" risk.
  Consider splitting into structured tools: assess_threat, assess_escape, assess_ability.
  Counterargument: more tools = more token cost. Decision needed.

- [ ] **Candidate score shown to LLM causes anchor bias**
  If fallback auto-picks highest score AND LLM sees scores, LLM will over-rely on scores.
  Options: (a) hide scores from LLM, (b) show only rank not score, (c) accept bias.

- [ ] **SafetyGate at execute-only is single-layer**
  Pro recommends pre-filter + post-validation (like Claude Code's permissions + sandbox).
  Consider: pre-filter on navigate_to (dangerous zone warning), pre-filter on choose
  (irreversible dialogue options).

- [ ] **Replay/eval protocol needs realistic design**
  CoQ seed investigation (2026-03-18) revealed:
  - Seed selection is currently disabled (RNG instability fix pending)
  - Even with same seed, combat RNG is a separate channel (not deterministic)
  - Enemy spawns depend on turn-of-entry, not seed alone
  Therefore:
  - "Fixed-seed identical replay" is NOT possible for full agent runs
  - C# component regression tests (recorded input → expected output) ARE possible
  - Agent improvement comparison must be statistical (N-run averages), not paired replay
  - Golden trace = C# unit test fixtures, not full-game replays
  API: `XRLCore.Core.Game.GetWorldSeed(null)`, `Stat.GetSeededRandomGenerator()`

- [ ] **Eval suite missing (self-improvement 5 conditions: 3/5 met)**
  Met: logging, versioning, death taxonomy.
  Missing: eval suite (fixed-seed replay), counterfactual logging.
  Without these, Appendix A cannot function.

- [ ] **Policy boundary unclear**
  Define explicitly: what LLM decides vs CandidateGenerator decides vs
  get_help advises vs SafetyGate overrides. Currently "a bit muddied."

- [ ] **"no dual-path" is misleading**
  Safe exploration via C# heuristic IS a separate path.
  Rename to "bounded autonomy region" and make explicit.

- [ ] **v2.1 references not self-contained**
  Plan references v2.1 for many core components but doesn't embed them.
  For self-contained spec: inline critical v2.1 sections or add explicit cross-refs.
