**1. Go/No-Go 判定**

**条件付き Go。Clean Go ではありません。**

Phase 0 着手自体は可能ですが、`architecture-v5.md` を v5.7 として小さく直してから implementation spec として使うべきです。A/B の実コード照合はほぼ通っていますが、B の exit criteria と C の fallback 記述にまだ実装者を誤誘導するズレがあります。

**2. MF-v5.5-A/B/C 解消状況**

- **MF-v5.5-A: ✅解消**
  `AutoAct.cs:465` の実 signature と v5.6 の `Type[]` は一致。
  順序も `GameObject Actor → Cell FromCell → ref GameObject LastDoor → Cell ToCell → string Direction → bool x5` で正しい。
  9引数 overload は `AutoAct.cs:613` から L616 で 10引数 overload に delegate するため、10引数版 patch で覆える。

- **MF-v5.5-B: ⚠️部分解消**
  loader flow は正しい (`BuildMods()` → `TryBuildAssembly()` → `CompileFromFiles()` → `ApplyHarmonyPatches()`)。
  ただし exit criteria の `Compiling N files...` は厳密には不正確。`ModInfo.cs:768-769` は 1ファイル時に `Compiling 1 file...`、複数時に `Compiling N files...` を出す。
  Phase 0-A2 の検収文字列としては `Compiling <N> file(s)...` または正規表現にすべき。

- **MF-v5.5-C: ⚠️部分解消**
  Matrix と `fallback_choice_id` semantics は大きく改善。
  ただし L214-216, L259-269 がまだ「unsupported modals escalate to supervisor」と読め、Matrix の「cancel_or_back first, then supervisor」と完全には収束していない。
  さらに `state_fallback()` は `fallback_choice_id` 失敗後に `cancel_or_back` を試す点は正しいが、両方失敗した後が L2074 で `Action.WAIT` になっている。modal で WAIT は deadlock 温存になり得るため、ここは `WAIT_FOR_SUPERVISOR` が正しい。

**3. 新規 Must-Fix**

- **MF-v5.6-1: 0-A2 build log exit criteria の singular/plural 修正**
  `Compiling N files...` を literal 判定にしない。`Compiling <N> file(s)...` に直す。

- **MF-v5.6-2: modal fallback の最後の逃げ先を supervisor に統一**
  unsupported modal: `cancel_or_back` if possible, else `supervisor_request`。
  dialogue/confirmation fallback が失敗し、`cancel_or_back` も失敗した場合も `WAIT` ではなく `WAIT_FOR_SUPERVISOR`。

`modal is None` 分岐は防御的処理として置いてよい。ただし意味は「unsupported modal」ではなく「state invariant violation / stale notification」。まず `cancel_or_back` を試す設計は許容できるが、失敗時は supervisor escalation が必要。

- **S1 duplicate guard**: Phase 0-A に入れる。Must 昇格不要。ただし mod reload smoke の acceptance に含めるべき。
- **S2 BackgroundClicked null-safe**: Phase 2 Gate 2 前。実コード `PopupMessage.cs:876-879` は `FirstOrDefault()` 後に null deref し得るので、直接 `BackgroundClicked()` 依存は避ける。
- **S3 TextConsole lock/ViewTag**: Phase 0-B。観測品質に直結するので早め。
- **S6 p95 数学**: Phase 2b の計測設計でよい。Must ではない。

Merchant が L717-718 と L1051-1052 に残るのは OK。そこは policy ではなく検出・cancel mapping の実装表。
