**1. Go/No-Go 判定**

**条件付き Go**。Phase 0 の探索実装は始めてよいです。ただし v5.4 は「redline complete / Phase 2a First Demo まで大きな手戻りなし」とまでは言えません。実コード照合で、核 API の多くは正しい一方、**Harmony patch 指定、モーダル操作の thread-affinity、level-up fallback、idempotency の Phase 配置、MOD packaging** は実装前に v5.5 へ赤入れすべきです。

Phase 0-A/B/C は並列ではなく、**0-A の MOD skeleton とロード確認が B/C の前提**です。

**2. Must-Fix**

1. **`AutoAct.TryToMove` Harmony patch 指定が危険**
   docs は「Postfix」と書くがコード例は `Prefix`。さらに `TryToMove` は overload が 2 つあり、素の `[HarmonyPatch(typeof(AutoAct), nameof(AutoAct.TryToMove))]` は曖昧です。対象は `ref GameObject LastDoor` 付き overload に明示すべき。
   根拠: `docs/architecture-v5.md:455`, `decompiled/XRL.World.Capabilities/AutoAct.cs:465`, `decompiled/XRL.World.Capabilities/AutoAct.cs:613`, `decompiled/XRL.Core/ActionManager.cs:1014`

2. **`IPlayerSystem` は正しいが、`RegisterPlayer()` 内の event 登録を仕様に明記不足**
   `BeginTakeActionEvent.Check()` は `Object.HandleEvent()` に配送されるので IGameSystem 単体不可、という v5.1 修正は正しい。ただし `RegisterPlayer()` で `SingletonEvent<BeginTakeActionEvent>.ID` を登録しなければ受けられません。
   根拠: `decompiled/XRL.World/BeginTakeActionEvent.cs:37`, `decompiled/XRL.World/BeginTakeActionEvent.cs:50`, `decompiled/XRL/IPlayerSystem.cs:35`, `decompiled/XRL/WanderSystem.cs:57`

3. **モーダル操作は gameQueue 一択では死ぬ**
   new popup は `uiQueue.awaitTask()` で UI に出し、game thread は `complete.Task.Wait()` で待ちます。WebSocket thread から `choose` / `cancel_or_back` を gameQueue に投げる設計だと、詰まる可能性が高い。PopupMessage 操作は UI thread 経由を明記すべきです。
   根拠: `decompiled/XRL.UI/Popup.cs:823`, `decompiled/XRL.UI/Popup.cs:908`, `decompiled/QupKit/ThreadTaskQueue.cs:135`, `decompiled/GameManager.cs:142`

4. **level-up fallback が仕様内で矛盾**
   本文では level-up は `fallback_choice_id=null → WAIT_FOR_SUPERVISOR`。しかし State × Fallback Matrix の WebSocket disconnect は「C# selects fallback_choice_id」のままです。disconnect 時の level-up は supervisor/paused/offline-safe policy を明記すべき。
   根拠: `docs/architecture-v5.md:208`, `docs/architecture-v5.md:607`, `decompiled/XRL.UI/StatusScreen.cs:691`, `decompiled/XRL.World.Parts.Mutation/UnstableGenome.cs:47`

5. **full idempotency を Phase 2b に送るのは遅い**
   v5.4 は `message_id/action_nonce/state_version/session_epoch` で duplicate/stale/reconnect を解く設計だが、実タスクでは full idempotency が Phase 2b。First Demo で LLM terminal action を流すなら、少なくとも terminal actions の `action_nonce + state_version + cached result` は Phase 2a Gate 1 前に必要。
   根拠: `docs/architecture-v5.md:2166`, `docs/architecture-v5.md:2438`, `docs/architecture-v5.md:2545`, `docs/architecture-v5.md:2619`

6. **MOD build/package の具体像がない**
   Phase 0-A は MOD skeleton なのに、`manifest.json`、assembly 配置、`.csproj`、HarmonyLib 参照、ロード確認手順が未定義。`ModInfo.ApplyHarmonyPatches()` は assembly に Harmony attribute がある場合だけ patch します。
   根拠: `decompiled/XRL/ModInfo.cs:847`, `decompiled/XRL/ModInfo.cs:855`

**5. コード実装時のピットフォール予想**

- AutoAct interrupt reason は本当に MOD synthesis 必須。`Interrupt()` は空 reason が多く、`AutomoveInterruptBecause` だけでは足りません。根拠: `decompiled/XRL.World.Capabilities/AutoAct.cs:391`, `decompiled/XRL.Core/ActionManager.cs:850`
- `ArePerceptibleHostilesNearby()` は `ExtraHostilePerceptionEvent` を含むので非視覚 hostile 対応は正しい。根拠: `decompiled/XRL.World/GameObject.cs:11138`, `decompiled/XRL.World/ExtraHostilePerceptionEvent.cs:35`
- `Cell.GetLocalAdjacentCells()` と `Cell.GetCover()` の前提は正しい。根拠: `decompiled/XRL.World/Cell.cs:286`, `decompiled/XRL.World/Cell.cs:6877`, `decompiled/XRL.World/Zone.cs:7790`
- C# / Python WS は「C# が Unity main/game/ui thread を持つ」点が一番汚くなります。tool handler ごとに `gameQueue` / `uiQueue` / background-only を分ける表が必要です。
- ability activation は Phase 0b で十分ではなく、target picker / confirmation popup / self-target confirm が絡みます。Warden first demo に絞る判断は妥当です。
- `agency_share_llm >60%` は First Demo だと厳しいです。AutoAct が長距離移動を担当するため、v5.3 の 2a criteria `>50%` の方が現実的です。

**6. v5.5 redline 案**

```diff
- The C# MOD uses a Harmony Postfix patch on AutoAct.TryToMove()
+ The C# MOD uses a Harmony Prefix patch on the ref-LastDoor overload of
+ AutoAct.TryToMove(..., ref GameObject LastDoor, ..., bool AllowDigging, ...).
+ The patch MUST specify argument types because TryToMove has overloads.
+ Prefix may declare `ref bool AllowDigging` and set it false when NavigateToActive.

- class LLMOfQudSystem : IPlayerSystem
+ class LLMOfQudSystem : IPlayerSystem
+ RegisterPlayer(Player, Registrar) MUST call:
+   Registrar.Register(SingletonEvent<BeginTakeActionEvent>.ID);
+ HandleEvent(BeginTakeActionEvent E) receives object-level player turn events.

- Cross-thread: use GameManager.Instance.gameQueue.queueTask(...)
+ Cross-thread routing:
+   game state reads / CommandEvent.Send -> gameQueue
+   PopupMessage OnSelect/OnActivateCommand / new UI modal interaction -> uiQueue
+   never call awaitTask from a thread whose target queue cannot be pumped.

- Modal WebSocket disconnected -> C# selects fallback_choice_id
+ Modal WebSocket disconnected:
+   dialogue/confirmation -> C# selects fallback_choice_id
+   level_up/build-altering modal -> pause + supervisor_request; no auto-select
+   unsupported modal -> cancel_or_back if safe, else supervisor_request

- Full idempotency remains in Phase 2b
+ Terminal-action idempotency (`action_nonce`, `state_version`, cached duplicate result)
+ is required before Phase 2a Gate 1. Phase 2b may harden reconnect and all non-terminal
+ tool deduplication.
```
