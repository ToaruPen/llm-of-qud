**1. Go/No-Go 判定**

**条件付き Go 継続。Clean Go ではありません。**
Phase 0 の 0-A 調査実装は始めてよいですが、**0-A2 packaging と M1 Harmony 指定は v5.6 redline を先に入れるべき**です。特に MOD packaging は Phase 0 の直近ブロッカーです。

**2. Must-Fix 解消状況**

1. **M1 AutoAct.TryToMove Harmony 指定: ❌未解消**
   ドキュメントの `Type[]` が実 signature と合っていません。
   実コードは `TryToMove(GameObject, Cell, ref GameObject, Cell, string, bool, bool, bool, bool, bool)` です。
   v5.5 の例は `typeof(bool)` が 3番目に入り、`Cell` / `string` も欠けています。

2. **M2 IPlayerSystem.RegisterPlayer event ID 登録: ✅解消**
   `Registrar.Register(SingletonEvent<BeginTakeActionEvent>.ID)` の形で正しいです。
   `BeginTakeActionEvent.Check()` は `Object.HandleEvent()` へ配送され、`IPlayerSystem.RegisterPlayer()` で player body 側に登録する設計と整合しています。

3. **M3 Cross-thread routing: ✅概ね解消**
   `gameQueue` / `uiQueue` / WebSocket thread の分離は妥当です。
   `ConversationUI.Select(int)` は static conversation state を触るため `gameQueue` 扱いでよいです。`CommandEvent.Send` も game-side 操作なので `gameQueue` が妥当です。

4. **M4 level-up / unsupported modal fallback: ⚠️部分解消**
   Matrix 本体は改善されていますが、周辺記述がまだ矛盾しています。
   `docs/architecture-v5.md:208` がまだ「C# selects fallback_choice_id」、`230` が「select default modal option」、`648` が merchant に `fallback_choice_id` を与える、と読めます。`state_fallback()` 疑似コードも unsupported modal を扱わず、`fallback_choice_id=null` で `choose` し得ます。

5. **M5 Terminal-action idempotency Phase 1 昇格: ✅解消**
   Phase 1-G が Phase 2a Gate 1 の hard prerequisite として読めます。
   ただし `cancel_or_back` は非-terminal だが状態変更するので、Gate 2 前の duplicate 対策は Should-Fix として残ります。

6. **M6 MOD build/package 手順: ⚠️部分解消**
   0-A2 を追加した点は正しいですが、内容が実 loader とずれています。
   decompiled では MOD は `.cs` を検出して Roslyn compile し、`TryBuildAssembly()` 成功後に `ApplyHarmonyPatches()` します。`.dll` は `ModFileType.Assembly` に分類されますが、提示範囲では prebuilt DLL をロードする経路が見えません。`manifest.json` に "entry assembly" field もありません。

- **S1 duplicate guard**: Phase 0 内で吸収可能。ただし 0-A の acceptance に入れるべき。
- **S2 BackgroundClicked null-safe**: Phase 2a Gate 2 前で十分。ただし `BackgroundClicked()` 直呼びは避け、cancel item lookup を明示。
- **S3 TextConsole.CurrentBuffer lock/ViewTag**: Phase 0-B で吸収すべき。観測精度に直結します。
- **S4 Phase 0-I 縮小**: 対応済み。
- **S5 force_action_tools 意味整合**: 対応済み。
- **S6 p95 5s 数学**: Must ではないが、KPI 章に「combat 4-call turns は p95 母集団を分ける」と明記済みなので Phase 2 で測定調整でよいです。

**5. v5.6 redline 案**

```diff
- new Type[] {
-     typeof(GameObject), typeof(Cell), typeof(bool),
-     typeof(GameObject).MakeByRefType(),
-     typeof(bool), typeof(bool), typeof(bool)
- }
+ new Type[] {
+     typeof(GameObject), typeof(Cell), typeof(GameObject).MakeByRefType(),
+     typeof(Cell), typeof(string),
+     typeof(bool), typeof(bool), typeof(bool), typeof(bool), typeof(bool)
+ }

- Modal: WebSocket disconnected -> C# selects fallback_choice_id
+ Modal: WebSocket disconnected ->
+   dialogue/confirmation: C# selects fallback_choice_id
+   level_up/build-altering: pause + supervisor_request
+   unsupported: cancel_or_back if safe, else supervisor_request

- Merchant: fallback_choice_id is "Leave" / Escape
+ Merchant/quantity/string/unsupported: fallback_choice_id is null in MVP;
+ fallback uses cancel_or_back if safe, else supervisor_request.

- manifest.json (name, version, entry assembly, description, compatibility flags)
- .csproj targeting the engine's .NET runtime; assembly path matches what ModInfo.LoadFromDirectory scans for
+ manifest.json using fields present in XRL.ModManifest: ID, Title, Description,
+ Version, Author, optional Directories/Dependencies.
+ CoQ scripting mods are compiled from .cs files discovered under the mod directory.
+ Do not assume prebuilt DLL loading unless verified against the installed binary.

+ Phase 0 exit criteria:
+ - MOD appears in CoQ active mod list, its .cs files compile, and the load probe log appears once.
```
